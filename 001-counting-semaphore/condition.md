# Модуль 001 — Counting Semaphore

## Контекст

Семафор — один из древнейших примитивов конкурентного программирования (Dijkstra, 1965). Его задача проста: ограничить количество горутин, одновременно выполняющих определённую операцию. 10 соединений к базе — семафор на 10. 5 одновременных HTTP запросов к внешнему API — семафор на 5.

В стандартной библиотеке Go семафора нет. Есть `golang.org/x/sync/semaphore`, но это внешний пакет. Семафор реализуется через буферизованный канал: ёмкость = количество разрешений, отправка = acquire, чтение = release. Это не хак — это идиоматичный Go: каналы предназначены для координации горутин.

В workerpool-practice каналы использовались для передачи задач и результатов. Здесь канал используется как счётчик: содержимое неважно (`struct{}`), важна ёмкость и текущая заполненность.

## Что нужно реализовать

Пакет `semaphore` с типом `Semaphore`.

**Тип `Semaphore`:**

```go
type Semaphore struct {
    ch chan struct{}
}
```

**Конструктор `New(maxConcurrency int) *Semaphore`** — создаёт семафор с указанным количеством разрешений. Паника если `maxConcurrency <= 0`.

**Методы:**

- `Acquire()` — блокируется до получения разрешения. Отправляет `struct{}{}` в канал. Если канал полон (все разрешения заняты) — горутина блокируется на отправке.

- `Release()` — возвращает разрешение. Читает из канала. Если канал пуст (Release без предшествующего Acquire) — горутина блокируется навечно (дедлок). Это баг вызывающего кода, не семафора.

- `TryAcquire() bool` — неблокирующая попытка. Возвращает `true` если разрешение получено, `false` если все заняты. Через `select` с `default`.

- `Available() int` — количество свободных разрешений: `cap(ch) - len(ch)`.

- `MaxConcurrency() int` — максимальное количество разрешений: `cap(ch)`.

**Файл `main.go`** — демонстрация:
- Семафор на 3 разрешения
- Запуск 10 горутин, каждая делает Acquire → работу (sleep) → Release
- Вывод показывает что одновременно работают максимум 3
- Демонстрация TryAcquire: при полном семафоре возвращает false
- Демонстрация Available: до и после Acquire

## Требования

1. `chan struct{}` — не `chan bool`, не `chan int`. `struct{}` — нулевой размер. Канал используется как счётчик, содержимое не несёт информации.

2. Семантика: `Acquire` = отправка в канал, `Release` = чтение из канала. Это инвертированная интуиция: "получить разрешение" = "положить токен", "вернуть разрешение" = "забрать токен". Логика: буферизованный канал с capacity N может принять N отправок до блокировки. Каждая отправка "занимает" один слот. Чтение освобождает слот.

3. `TryAcquire` через `select` с `default` — стандартный паттерн неблокирующей операции с каналом:
   ```go
   select {
   case ch <- struct{}{}:
       return true
   default:
       return false
   }
   ```

4. `Release` без предшествующего `Acquire` — чтение из пустого канала — вечная блокировка. Это ответственность вызывающего кода. Семафор не считает кто сделал Acquire — у него нет понятия "владения". Это отличие от мьютекса (где unlock без lock — паника в runtime).

5. `Available()` использует `len(ch)` и `cap(ch)`. `len` на канале — количество элементов в буфере. `cap` — ёмкость буфера. Оба — O(1), потокобезопасны, не требуют синхронизации.

6. Race detector: `go run -race main.go` обязателен. Семафор — примитив синхронизации, он ДОЛЖЕН быть race-free.

## Эксперимент на разрушение

**Эксперимент 1: Release без Acquire.** Создать семафор на 3. Не делать Acquire. Вызвать Release. Зафиксировать: горутина заблокирована навечно (дедлок). С одной горутиной — `fatal error: all goroutines are asleep - deadlock!`. Объяснить: чтение из пустого буферизованного канала блокируется. У семафора нет защиты от этого — ответственность на вызывающем коде. Сравнить с `sync.Mutex`: `Unlock` без `Lock` — `fatal error: sync: unlock of unlocked mutex` (runtime проверяет).

**Эксперимент 2: Больше Release чем Acquire.** Сделать 1 Acquire, затем 2 Release. Первый Release — успешен. Второй — блокировка (или дедлок). Зафиксировать: семафор не считает balance. Если такой дисбаланс происходит в production — ресурс "утекает" (один лишний Release делает Available > MaxConcurrency при следующем цикле). Написать тест который обнаруживает дисбаланс через `Available() > MaxConcurrency()`.

**Эксперимент 3: Наблюдение за конкурентностью.** Запустить 100 горутин с семафором на 5. Каждая горутина инкрементирует `atomic.Int64` при Acquire и декрементирует при Release. В отдельной горутине-наблюдателе каждые 10ms читать значение счётчика и записывать максимум. Убедиться что максимум ≤ 5. Затем убрать семафор — максимум стремится к 100. Зафиксировать: семафор гарантирует верхнюю границу конкурентности.

## Вопросы до написания кода

1. Почему семафор реализуется через `chan struct{}` а не через `sync.Mutex` + счётчик? Какие свойства каналов делают их естественным выбором для семафора?

2. `Acquire` = отправка в канал, `Release` = чтение из канала. Это инвертированная интуиция (обычно "получить" = "прочитать"). Почему именно так? Что произойдёт если поменять местами (Acquire = чтение, Release = отправка)?

3. `TryAcquire` использует `select` с `default`. Почему `default` делает операцию неблокирующей? Что происходит в runtime когда выполняется `select` с `default`?

4. `Available()` возвращает `cap(ch) - len(ch)`. Является ли это значение точным в конкурентном контексте? Может ли значение устареть к моменту использования?

5. Семафор на 1 (`New(1)`) — это мьютекс? В чём отличие от `sync.Mutex`? (Подсказка: владение, повторный lock, runtime проверки.)

6. `struct{}{}` — два набора фигурных скобок. Первый — тип (`struct{}`), второй — литерал значения. Какой размер у `struct{}` в памяти? Почему это важно для канала-семафора?
