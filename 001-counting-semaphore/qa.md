# Модуль 001 — QA: Counting Semaphore

## Блок 1: Внутренний механизм канала-семафора

**Q1.1:** Буферизованный канал `make(chan struct{}, 5)` — что именно аллоцируется в памяти? Опиши структуру `hchan` (runtime/chan.go): где лежит буфер, что такое `sendq` и `recvq`, какую роль играет `lock` внутри канала. Почему канал потокобезопасен без внешней синхронизации?

**A1.1:** При `make(chan struct{}, 5)` аллоцируется структура `hchan`: кольцевой буфер (для `struct{}` — формально нулевого размера, но слоты учитываются), мьютекс `lock` (не `sync.Mutex`, а внутренний `runtime.mutex` — более лёгкий), два списка ожидающих горутин (`sendq` для заблокированных на отправке, `recvq` для заблокированных на чтении), счётчики `qcount` (текущая заполненность) и `dataqsiz` (ёмкость). Каждая операция send/receive захватывает внутренний lock канала. Потокобезопасность гарантируется runtime — разработчику не нужно думать о синхронизации доступа к каналу.

**Q1.2:** Горутина вызывает `Acquire()` (send в канал) когда буфер полон. Что происходит на уровне Go scheduler? Где горутина "ждёт"? Что её разбудит?

**A1.2:** Горутина создаёт `sudog` — структуру описывающую ожидающую горутину. `sudog` добавляется в `sendq` канала. Горутина переводится в состояние `Gwaiting` — она снимается с OS-потока (M) и помещается в очередь ожидания канала. OS-поток освобождается для выполнения других горутин. Когда другая горутина вызывает `Release()` (receive), runtime извлекает первый `sudog` из `sendq`, переводит горутину в `Grunnable` и ставит в run queue процессора (P). Scheduler подхватит её при следующем scheduling tick. Важно: ожидание на канале НЕ занимает OS-поток — это одно из ключевых свойств горутин.

**Q1.3:** `len(ch)` на буферизованном канале — runtime читает `hchan.qcount`. Это поле защищено мьютексом канала. Означает ли это что `len(ch)` захватывает lock? Если да — какой overhead? Если нет — как обеспечивается корректность?

**A1.3:** В текущей реализации Go runtime `len(ch)` действительно захватывает внутренний lock канала на очень короткое время (чтение одного uint). Overhead минимален — внутренний `runtime.mutex` значительно легче `sync.Mutex` (использует futex/semaphore ОС напрямую, без fairness и starvation prevention). Но это означает что частый вызов `len(ch)` в горячем цикле может создавать contention на lock канала, конкурируя с send/receive операциями. Для метрик — допустимо. Для высокочастотного polling — нежелательно.

## Блок 2: Семантика и контракт

**Q2.1:** Семафор на 3. Горутина A сделала Acquire. Горутина B делает Release (не A). Валидно ли это? Что произойдёт? Чем это отличается от поведения `sync.Mutex`?

**A2.1:** Для семафора это валидно — семафор не имеет понятия владения. Горутина B успешно прочитает из канала, увеличив количество свободных разрешений. Это означает что семафор может быть "разблокирован" другой горутиной — корректное поведение для producer-consumer паттерна. Для `sync.Mutex` это невалидно: начиная с Go 1.8+ `Unlock` из горутины отличной от той что сделала `Lock` вызывает `fatal error: sync: unlock of unlocked mutex` (хотя формально это не гарантировано спецификацией, runtime это проверяет). Отсутствие владения у семафора — одновременно сила (гибкость) и слабость (нет protection от неправильного использования).

**Q2.2:** Семафор на 5. Код:
```go
sem.Acquire()
sem.Acquire()
defer sem.Release()
defer sem.Release()
```
Одна горутина. Сколько разрешений занято? Когда они освободятся? Есть ли проблема?

**A2.2:** Занято 2 разрешения. Оба Release выполнятся при выходе из функции (defer — LIFO порядок, но для двух одинаковых Release порядок не важен). Проблемы нет — семафор на 5 позволяет одной горутине сделать до 5 Acquire подряд. Это отличается от мьютекса: повторный `Lock()` из той же горутины — дедлок (mutex в Go не рекурсивен). Семафор "не знает" кто делает Acquire — для него это просто две отправки в канал.

**Q2.3:** `TryAcquire` вернул `true`. Между `TryAcquire` и `defer Release()` произошла паника. Будет ли утечка разрешения?

**A2.3:** Зависит от того, выполнился ли `defer` до паники. Если `defer sem.Release()` стоит ДО паники — `defer` выполнится при раскрутке стека (паника вызывает выполнение всех deferred функций текущей горутины). Если `defer` не успел зарегистрироваться (паника произошла между `TryAcquire` и `defer`):
```go
if sem.TryAcquire() {
    panic("oops")      // defer ещё не зарегистрирован!
    defer sem.Release() // этот код не достигнут
}
```
В этом случае — утечка разрешения. Правильный порядок:
```go
if sem.TryAcquire() {
    defer sem.Release() // СРАЗУ после acquire
    panic("oops")       // defer уже зарегистрирован, выполнится
}
```

## Блок 3: Edge cases и подводные камни

**Q3.1:** `New(0)` — семафор на 0 разрешений. Канал `make(chan struct{}, 0)` — небуферизованный. Что произойдёт при `Acquire()`? При `TryAcquire()`? Имеет ли смысл семафор на 0?

**A3.1:** `Acquire()` на небуферизованном канале блокируется навсегда (отправка в небуферизованный канал блокируется пока кто-то не читает). `TryAcquire()` всегда вернёт `false` (буфер нулевой, отправка невозможна без получателя). Семафор на 0 не имеет практического смысла — никто не может получить разрешение. Поэтому конструктор должен паниковать при `maxConcurrency <= 0`. Паника при невалидном параметре — конвенция Go для програмных ошибок (аналог `make([]int, -1)` — тоже паника).

**Q3.2:** Горутина сделала `Acquire()`, затем была принудительно остановлена (например, через `runtime.Goexit()`). Разрешение утечёт навсегда?

**A3.2:** `runtime.Goexit()` вызывает выполнение всех deferred функций текущей горутины, затем завершает горутину. Если `defer sem.Release()` зарегистрирован — разрешение вернётся. Если defer нет — да, утечка навсегда. Канал не имеет финализатора по горутинам. Это ещё один аргумент за `defer sem.Release()` сразу после Acquire.

**Q3.3:** Семафор на 1000. В пике одновременно работают 1000 горутин, каждая держит разрешение 10 секунд. Сколько памяти занимает семафор (канал)? А сколько — 1000 горутин?

**A3.3:** Канал `make(chan struct{}, 1000)` — структура `hchan` + буфер. Для `struct{}` буфер формально 0 байт данных, но runtime всё равно аллоцирует слоты для учёта (каждый слот — несколько байт метаданных). Грубо: сотни байт — килобайт. Это ничтожно. 1000 горутин — каждая горутина стартует с ~2-8KB стека (зависит от версии Go), итого ~2-8MB. Стек может расти до мегабайтов при глубоком call stack. Семафор — не bottleneck по памяти. Горутины — основной потребитель.

**Q3.4:** `Available()` вызывается из горутины-наблюдателя каждые 100ms для записи метрик. Одновременно 50 горутин делают Acquire/Release. Является ли `Available()` потокобезопасным? Может ли он вернуть отрицательное значение? Может ли вернуть значение больше `MaxConcurrency()`?

**A3.4:** `cap(ch)` и `len(ch)` потокобезопасны — runtime гарантирует это. `Available() = cap(ch) - len(ch)` вычисляется в два шага, и между `cap` и `len` другие горутины могут модифицировать канал. Отрицательное значение невозможно: `len(ch)` ≤ `cap(ch)` гарантировано runtime. Значение больше `MaxConcurrency()` тоже невозможно при корректном использовании: `len(ch) ≥ 0`, значит `cap - len ≤ cap`. НО: при дисбалансе Release > Acquire (баг вызывающего кода) — `len(ch)` может быть меньше ожидаемого, что даст `Available()` > реального числа свободных разрешений. Семафор не защищает от некорректного использования.

## Блок 4: Сравнения и альтернативы

**Q4.1:** `golang.org/x/sync/semaphore.Weighted` использует `sync.Mutex` + `sync.Cond` + linked list внутри. Канал-семафор использует `chan struct{}`. Назвать минимум три конкретных отличия в поведении и производительности.

**A4.1:** (1) **Weighted acquire**: `x/sync/semaphore` поддерживает `Acquire(ctx, n)` — забрать n разрешений атомарно. Канал не поддерживает "отправить n элементов атомарно" — нужен цикл из n send, что не атомарно (горутина может получить 3 из 5 запрошенных и заблокироваться на 4-м). (2) **Fairness/FIFO**: `x/sync/semaphore` поддерживает FIFO — запросы обрабатываются в порядке поступления. Канал с несколькими ожидающими — runtime выбирает по своему усмотрению (зависит от scheduler, формально не FIFO). (3) **Cancel**: `x/sync/semaphore.Acquire(ctx, n)` отменяется через context. Для канала нужен `select` с `ctx.Done()` (модуль 002). (4) **Overhead**: для unweighted семафора канал быстрее — send/receive оптимизированы в runtime. Для weighted — `x/sync` эффективнее (нет цикла из n операций).

**Q4.2:** WaitGroup vs семафор. Оба ограничивают конкурентность. `wg.Add(n)` + `wg.Wait()` vs `sem.Acquire()` + `sem.Release()`. В чём фундаментальная разница в семантике?

**A4.2:** `WaitGroup` — барьер ожидания: "подожди пока N операций завершатся". Не ограничивает одновременность — все N могут работать параллельно. Семафор — ограничитель одновременности: "не более N одновременно". WaitGroup считает завершения, семафор считает активных. WaitGroup — fan-out + wait-for-all. Семафор — throttling. Они комплементарны: WaitGroup используется чтобы дождаться всех горутин, семафор — чтобы не запустить слишком много одновременно.

**Q4.3:** `make(chan struct{}, N)` и `make(chan struct{}, 1)` использованный N раз (N семафоров по 1). Дают ли они одинаковый эффект? Могут ли N бинарных семафоров заменить один counting семафор на N?

**A4.3:** Нет. Counting семафор на N позволяет одной горутине получить все N разрешений. N бинарных семафоров — горутина должна знать какой именно семафор свободен, и acquire N разных семафоров не атомарен (после получения 5 из 10 может заблокироваться на 6-м, удерживая 5 — потенциальный deadlock при взаимном ожидании с другой горутиной). Counting семафор — единая очередь, N бинарных — N очередей. Поведение под нагрузкой принципиально разное.

## Блок 5: Production-сценарии

**Q5.1:** HTTP сервер, семафор ограничивает конкурентные запросы к базе данных до 10. Под нагрузкой семафор полностью заполнен. Новый запрос вызывает `Acquire()`. Клиент отключается через 5 секунд (TCP timeout). Горутина handler всё ещё заблокирована на `Acquire`. Что произойдёт? Как это решить?

**A5.1:** Горутина останется заблокированной на `Acquire()` навсегда — даже после отключения клиента. `http.ResponseWriter` закроется, но горутина не узнает об этом потому что блокируется на канале, а не на IO. Утечка горутин: каждый отключившийся клиент оставляет зависшую горутину. Решение — `AcquireWithContext(r.Context())` (модуль 002): context запроса отменяется при отключении клиента, select с ctx.Done() разблокирует горутину.

**Q5.2:** Семафор на 10. Под нагрузкой пиковая очередь ожидающих — 500 горутин. Всё работает. Затем нагрузка падает, но 3 горутины навсегда застряли (bug: пропущен Release из-за нелинейного control flow). Теперь доступно только 7 слотов вместо 10. Как обнаружить такую утечку?

**A5.2:** Метрики: периодически сравнивать `Available()` с ожидаемым значением. Если нагрузка нулевая а `Available() < MaxConcurrency()` — утечка. Конкретнее: горутина-монитор проверяет: если за последние 60 секунд не было Acquire, но `Available() != MaxConcurrency()` — алерт. Также: `runtime.NumGoroutine()` — если после завершения нагрузки количество горутин не возвращается к baseline — утечка горутин (и вероятно — утечка разрешений семафора). В production: pprof goroutine dump покажет где именно горутины заблокированы.

**Q5.3:** Семафор ограничивает конкурентность до 10. Все 10 слотов заняты операциями которые в среднем длятся 100ms. Приходит 1000 запросов в секунду. Какова средняя латентность ожидания в очереди? Когда система перестаёт справляться?

**A5.3:** Throughput семафора: 10 слотов × (1000ms / 100ms) = 100 операций/секунду. При 1000 req/sec — система не справляется, очередь растёт бесконечно (arrival rate > service rate). Средняя латентность растёт линейно со временем (модель M/M/c из теории очередей). Система перестаёт справляться когда arrival rate > max throughput. Решения: увеличить семафор (больше одновременных операций), уменьшить время операции (оптимизация), отклонять лишние запросы (rate limiting — модуль 005), масштабировать горизонтально.

## Блок 6: Код и реализация

**Q6.1:** Написать `Acquire` и `Release` для обеих реализаций семафора (acquire=send и acquire=receive). Для каждой: показать что произойдёт при double-Release.

**A6.1:**
Реализация 1 (acquire=send):
```go
func (s *Semaphore) Acquire() { s.ch <- struct{}{} }
func (s *Semaphore) Release() { <-s.ch }
```
Double-Release: два чтения из канала. Если канал пуст после первого — второе блокируется навечно.

Реализация 2 (acquire=receive, предзаполненная):
```go
func (s *Semaphore) Acquire() { <-s.ch }
func (s *Semaphore) Release() { s.ch <- struct{}{} }
```
Double-Release: два send в канал. Если после первого буфер полон — второй блокируется навечно. Но если буфер не полон (другие acquire активны) — silent corruption: `Available()` будет больше `MaxConcurrency()`, в системе появится "лишнее" разрешение.

**Q6.2:** Семафор используется в тесте:
```go
func TestConcurrency(t *testing.T) {
    sem := New(3)
    var active atomic.Int64
    var maxSeen atomic.Int64
    
    var wg sync.WaitGroup
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            sem.Acquire()
            defer sem.Release()
            
            cur := active.Add(1)
            // обновить maxSeen
            // ... работа ...
            active.Add(-1)
        }()
    }
    wg.Wait()
    
    if maxSeen.Load() > 3 {
        t.Errorf("max concurrent %d > 3", maxSeen.Load())
    }
}
```
Код обновления `maxSeen` отсутствует. Написать его. Почему наивное `if cur > maxSeen { maxSeen = cur }` некорректно? Как сделать корректно через `atomic`?

**A6.2:** Наивное `if cur > maxSeen.Load() { maxSeen.Store(cur) }` — race condition: между Load и Store другая горутина может записать большее значение, и текущая горутина перезапишет его меньшим. Корректное решение — CAS loop:
```go
for {
    old := maxSeen.Load()
    if cur <= old {
        break  // текущее значение не больше уже записанного
    }
    if maxSeen.CompareAndSwap(old, cur) {
        break  // успешно обновили
    }
    // CAS failed — другая горутина обновила раньше, попробовать снова
}
```
Или проще — всегда записывать и проверять в конце:
```go
// Каждая горутина записывает свой максимум
// В конце — один Load даёт глобальный максимум
```
Самый простой корректный вариант: после `active.Add(1)` всегда пытаться обновить через CAS. Или — записывать все значения `cur` в слайс (с мьютексом) и найти максимум после `wg.Wait()`.

**Q6.3:** Семафор создаётся в init() пакета:
```go
var globalSem = semaphore.New(10)
```
Три разных пакета импортируют этот пакет и используют `globalSem`. Корректно ли это? Какие проблемы создаёт глобальный семафор?

**A6.3:** Технически корректно — семафор потокобезопасен. Проблемы: (1) **Неявная зависимость**: пакеты зависят от глобального состояния, это затрудняет тестирование (нельзя создать изолированный семафор для теста). (2) **Hardcoded limit**: 10 — константа при компиляции, не настраивается. (3) **Shared fate**: если один пакет утечёт разрешение — страдают все три. (4) **Отсутствие контекста**: по `pprof` стектрейсу непонятно какой из трёх пакетов держит разрешения. Правильный подход — dependency injection: создать семафор в main, передать через конструктор каждому пакету.
