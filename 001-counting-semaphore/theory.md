# Модуль 001 — Theory: Семафор через буферизованный канал

## Семафор как абстракция

### Определение

Семафор — это целочисленный счётчик с двумя атомарными операциями:
- **P (acquire / wait / down)** — уменьшить счётчик на 1. Если счётчик = 0, блокироваться до тех пор пока кто-то не увеличит.
- **V (release / signal / up)** — увеличить счётчик на 1. Если кто-то заблокирован на P — разблокировать.

Counting semaphore — семафор с начальным значением > 1. Binary semaphore — с начальным значением 1 (аналог мьютекса, но без понятия владения).

Семафор не знает кто сделал P и кто сделал V. Одна горутина может сделать P, другая — V. Это принципиальное отличие от мьютекса, где Lock и Unlock должны выполняться одной и той же горутиной (Go runtime проверяет это для `sync.Mutex` начиная с Go 1.0).

### Применения

1. **Ограничение конкурентности.** Семафор на N — максимум N горутин в критической секции. Пример: не более 10 одновременных запросов к базе данных.

2. **Producer-consumer сигнализация.** Семафор начальным значением 0 — producer делает V (сигналит что данные готовы), consumer делает P (ждёт данные). Это фактически то что делает небуферизованный канал.

3. **Управление ресурсами.** Пул из N соединений — семафор на N контролирует что не создано больше N одновременно.

## Буферизованный канал как семафор

### Почему канал

В Go конкурентные примитивы строятся из каналов и мьютексов. Семафор можно реализовать через оба:

**Через мьютекс + счётчик + condition variable:**
```go
type Semaphore struct {
    mu    sync.Mutex
    cond  *sync.Cond
    count int
    max   int
}
```

Это работает, но требует ручного управления condition variable (Cond.Wait, Cond.Signal, Cond.Broadcast), что error-prone и verbose.

**Через буферизованный канал:**
```go
type Semaphore struct {
    ch chan struct{}
}
```

Канал с буфером размером N может принять N отправок без блокировки. N+1-я отправка блокируется. Это именно семантика семафора: N разрешений, N+1-й ждёт.

Канал проще, идиоматичнее для Go, и корректен по построению: runtime гарантирует потокобезопасность канала, нет необходимости в ручной синхронизации.

### Маппинг операций

```
Семафор         │  Канал
────────────────┼──────────────────
P (acquire)     │  ch <- struct{}{}   (отправка — занять слот)
V (release)     │  <-ch               (чтение — освободить слот)
count           │  cap(ch) - len(ch)  (свободные слоты)
max             │  cap(ch)            (ёмкость буфера)
```

### Инвертированная интуиция

Интуитивно "получить разрешение" = "взять что-то из пула" = "прочитать из канала". Но в канале-семафоре — наоборот: acquire = отправка, release = чтение.

Причина: буферизованный канал блокирует отправителя когда буфер полон. Это точная семантика: "когда все разрешения заняты — новый запрос ждёт". Если бы acquire = чтение, нужно было бы предзаполнить канал токенами при создании, и release = отправка. Это тоже работает:

```go
// Альтернативная реализация: предзаполненный канал
func New(n int) *Semaphore {
    ch := make(chan struct{}, n)
    for i := 0; i < n; i++ {
        ch <- struct{}{}  // предзаполнить токенами
    }
    return &Semaphore{ch: ch}
}

func (s *Semaphore) Acquire() { <-s.ch }          // взять токен
func (s *Semaphore) Release() { s.ch <- struct{}{} } // вернуть токен
```

Обе реализации корректны. Первая (acquire = send) чуть проще: не нужно предзаполнять канал при создании. Вторая (acquire = receive) ближе к интуиции "взять из пула". Для этого проекта используется первая — она более распространена в Go community.

## struct{}: нулевой тип

### Размер

`struct{}` — пустая структура. Её размер — 0 байтов:

```go
fmt.Println(unsafe.Sizeof(struct{}{}))  // 0
```

Канал `chan struct{}` хранит только метаданные синхронизации (очередь ожидающих, состояние), но не данные элементов. Для семафора данные не нужны — важен только факт отправки/получения.

### Почему не bool или int

`chan bool` или `chan int` технически работают, но:
- Тратят 1–8 байтов на каждый элемент буфера без необходимости
- Создают вопрос "какое значение отправлять?" (`true`? `42`?) — бессмысленный выбор
- `struct{}{}` явно говорит: "содержимое не важно, это сигнал"

`chan struct{}` — конвенция Go для сигнальных каналов. Встречается повсеместно: done-каналы, семафоры, нотификации.

## Неблокирующие операции: select с default

### Механизм

`select` с веткой `default` — неблокирующая операция:

```go
select {
case ch <- struct{}{}:
    // отправка успешна
default:
    // канал полон, отправка невозможна прямо сейчас
}
```

Без `default`: `select` блокируется пока одна из веток не станет готовой. С `default`: если ни одна ветка не готова — немедленно выполняется `default`.

Это прямой аналог `tryLock` в других языках: попытаться получить ресурс, но не ждать если недоступен.

### Runtime реализация

Когда Go runtime выполняет `select`:

1. Проверяет все ветки на готовность (канал не полон для send, не пуст для receive)
2. Если одна или несколько готовы — выбирает случайную (для fairness)
3. Если ни одна не готова И есть `default` — выполняет `default`
4. Если ни одна не готова И нет `default` — блокирует горутину, ставит в очереди всех каналов

Для `TryAcquire` шаг 3 — ключевой: немедленный возврат без блокировки.

### TryAcquire: когда использовать

`TryAcquire` полезен когда есть альтернативный путь:

```go
if sem.TryAcquire() {
    defer sem.Release()
    processInternally()
} else {
    // fallback: поставить в очередь, вернуть ошибку, перенаправить
    return ErrBusy
}
```

В production это паттерн для graceful degradation: если основной ресурс перегружен — не ждать, а немедленно отклонить запрос с кодом 503 (Service Unavailable) или перенаправить на fallback.

## len и cap на каналах

### Семантика

`len(ch)` — количество элементов в буфере канала. Для небуферизованного канала — всегда 0. Для буферизованного — текущая заполненность.

`cap(ch)` — ёмкость буфера. Для небуферизованного — 0. Для буферизованного — значение переданное в `make`.

Оба — O(1), потокобезопасны (runtime использует атомарные операции для чтения длины и ёмкости).

### Точность в конкурентном контексте

`Available()` возвращает `cap(ch) - len(ch)`. Это значение точно в момент вычисления, но может устареть к моменту использования:

```go
if sem.Available() > 0 {
    // Между проверкой и Acquire другая горутина может забрать разрешение
    sem.Acquire()  // может заблокироваться!
}
```

Это классическая TOCTOU (time-of-check-to-time-of-use) проблема. `Available()` полезен для метрик и мониторинга, но НЕ для принятия решений о синхронизации. Для условного acquire — использовать `TryAcquire`.

## Семафор на 1: binary semaphore vs mutex

### Сходства

Binary semaphore (capacity = 1) и `sync.Mutex` оба обеспечивают взаимное исключение: в критической секции — не более одной горутины.

```go
// Binary semaphore
sem := semaphore.New(1)
sem.Acquire()
// критическая секция
sem.Release()

// Mutex
var mu sync.Mutex
mu.Lock()
// критическая секция
mu.Unlock()
```

### Различия

1. **Владение.** `sync.Mutex` привязан к горутине: Lock и Unlock должна вызвать одна и та же горутина. Runtime проверяет это (с Go 1.0, усилено в Go 1.8+: `Unlock` из другой горутины — `fatal error`). Семафор не имеет понятия владения: Acquire в одной горутине, Release в другой — валидно.

2. **Повторный lock.** `sync.Mutex` не рекурсивен: повторный Lock из той же горутины — дедлок. Семафор на 1 — тоже дедлок при повторном Acquire из той же горутины (канал полон, отправка блокируется).

3. **Runtime проверки.** `sync.Mutex.Unlock()` без `Lock()` — `fatal error: sync: unlock of unlocked mutex`. Семафор `Release()` без `Acquire()` — вечная блокировка (чтение из пустого канала). Ошибка мьютекса обнаруживается немедленно с чётким сообщением. Ошибка семафора — тихий дедлок.

4. **Производительность.** `sync.Mutex` оптимизирован для короткодержащих критических секций (fast path через atomic). Канал-семафор — overhead runtime (scheduler, goroutine parking). Для защиты shared state — мьютекс быстрее. Для ограничения конкурентности — семафор семантически правильнее.

## Паттерн defer для Release

### Обязательность

`defer sem.Release()` сразу после `Acquire()` — обязательный паттерн:

```go
sem.Acquire()
defer sem.Release()
// ... работа которая может паниковать, возвращать ошибку, etc.
```

Без `defer`: ранний `return`, паника, или забытый `Release()` — утечка разрешений. Семафор постепенно исчерпывается, система зависает. Это аналог `defer f.Close()` для файлов (модуль 008 проекта 1) и `defer wg.Done()` для WaitGroup (workerpool-practice).

### Исключение

`defer` не подходит когда Release должен произойти не при выходе из функции, а раньше — например, после отправки результата в канал:

```go
sem.Acquire()
result := heavyComputation()
sem.Release()  // освободить слот ДО блокирующей отправки результата
resultCh <- result  // может заблокироваться — но слот уже свободен
```

В этом случае `defer` задержал бы Release до выхода из функции, включая время ожидания на `resultCh`. Семафор был бы занят дольше необходимого.

1. Семафор создан через `make(chan struct{}, 10)`. Что вернёт `len(ch)` сразу после создания? А после 3 вызовов `Acquire`? Как вычислить количество свободных разрешений?

2. Две горутины: A делает `Acquire`, B делает `Release`. Порядок неизвестен. Если B выполнит Release первой — что произойдёт? Зависит ли это от того буферизованный канал или нет?

3. `select` без `default` блокируется. `select` с `default` — нет. Что произойдёт с `select` у которого ВСЕ ветки готовы одновременно? Какая будет выбрана?

4. Семафор на 1000 — `make(chan struct{}, 1000)`. Сколько памяти это занимает? Зависит ли от размера буфера если элементы — `struct{}`?

5. `Available()` вернул 5. Между `Available()` и `Acquire()` другая горутина сделала 5 Acquire. Что произойдёт с текущей горутиной при вызове `Acquire`? Как называется эта проблема?
