# Проект 2 — Semaphore & Sync Primitives

## Что это

Connection Pool Manager с rate limiting, bulkhead-изоляцией и защитой от thundering herd. Универсальный компонент который управляет доступом к ограниченным ресурсам: соединениям с базой данных, внешним API, файловым дескрипторам.

```go
pool := connpool.New(
    connpool.WithMaxConns(10),
    connpool.WithRateLimit(100),          // 100 req/sec
    connpool.WithBulkhead("critical", 5), // 5 слотов для критичного трафика
)

conn, err := pool.Acquire(ctx)
if err != nil {
    // таймаут, пул закрыт, rate limit
}
defer conn.Release()
```

## Зачем именно этот проект

В стандартной библиотеке Go нет типа `Semaphore`. Нет `RateLimiter`. Нет `Bulkhead`. Все эти примитивы реализуются через каналы, мьютексы и атомики — инструменты которые уже знакомы из workerpool-practice. Но знакомство с инструментом и умение построить из него production-примитив — разные вещи.

Проект строит один продукт модуль за модулем. Каждый модуль добавляет один примитив синхронизации. К финальному модулю — полноценный менеджер соединений, который можно подключить к любому из последующих проектов как компонент.

## Связь с предыдущими проектами

Проект 1 (Metrics Exporter) отработал типовую систему Go, интерфейсы, IO и HTTP — фундамент языка. Этот проект возвращается к конкурентности: горутины, каналы, мьютексы, атомики из workerpool-practice, но теперь в контексте production-примитивов синхронизации. Паттерны из обоих предыдущих работ объединяются: интерфейсы для абстракций, замыкания для конфигурации, кастомные ошибки для обработки отказов, structured logging для наблюдаемости.

## Модульная структура

### 001 — Counting Semaphore

**Концепт:** семафор через буферизованный канал `chan struct{}`.

Семафор — это счётчик разрешений. `Acquire` забирает одно разрешение (блокируется если все заняты). `Release` возвращает разрешение. В Go семафор реализуется через буферизованный канал: ёмкость канала = количество разрешений, отправка в канал = acquire, чтение из канала = release. Это фундаментальный паттерн — из него строятся все последующие модули.

**Результат:** тип `Semaphore` с методами `Acquire()`, `Release()`, `TryAcquire() bool`.

---

### 002 — Context-Aware Semaphore

**Концепт:** `AcquireWithContext(ctx)` — acquire с отменой через context, таймаутом и дедлайном.

Блокирующий `Acquire` опасен в production: если все разрешения заняты — горутина висит вечно. `AcquireWithContext` использует `select` с `ctx.Done()` для ограничения времени ожидания. Это прямое применение паттернов из workerpool-practice (модуль 004 — Context & Select), но теперь в контексте синхронизационного примитива.

**Результат:** `Semaphore` с `AcquireWithContext(ctx context.Context) error`.

---

### 003 — Weighted Semaphore

**Концепт:** семафор с весами — одна операция может забрать несколько разрешений.

Не все операции равны. Лёгкий запрос к кэшу — 1 слот. Тяжёлый запрос с JOIN на пять таблиц — 3 слота. Weighted semaphore позволяет задать "вес" каждого acquire. Реализация сложнее простого канала: нужна очередь ожидающих с разными весами и fairness — большой запрос не должен голодать пока мелкие проскакивают.

**Результат:** `WeightedSemaphore` с `Acquire(ctx, weight int64) error`, `Release(weight int64)`.

---

### 004 — Mutex Patterns & sync.RWMutex

**Концепт:** мьютексы, RWMutex для read-heavy workloads, типичные ошибки с мьютексами.

Канал-семафор хорош для ограничения конкурентности. Мьютекс — для защиты shared state. `sync.Mutex` — exclusive lock. `sync.RWMutex` — множество читателей, один писатель. Этот модуль строит потокобезопасный кэш соединений: многие горутины читают состояние, редкие — модифицируют. RWMutex даёт параллельное чтение без блокировки.

**Результат:** `SafeMap[K, V]` — потокобезопасная обёртка с `RWMutex`, основа для пула соединений.

---

### 005 — Token Bucket Rate Limiter

**Концепт:** rate limiting через token bucket алгоритм.

Rate limiter ограничивает частоту операций: не более N запросов в секунду. Token bucket — классический алгоритм: ведро с токенами, каждый запрос забирает один токен, токены добавляются с фиксированной скоростью. Если ведро пусто — запрос ждёт или отклоняется. Реализация через `time.Ticker` для пополнения и канал для хранения токенов.

**Результат:** `RateLimiter` с `Allow() bool`, `Wait(ctx) error`, `Limit() rate`, `Burst() int`.

---

### 006 — Connection Pool Core

**Концепт:** пул соединений с acquire/release, idle management, lifetime.

Центральный модуль проекта. Пул управляет набором соединений: создаёт новые когда нужно (до лимита), переиспользует idle, закрывает просроченные. Это объединение семафора (ограничение количества), мьютекса (защита состояния пула), и управления жизненным циклом ресурса.

**Результат:** `Pool` с `Acquire(ctx) (*Conn, error)`, `Release(conn)`, `Close()`. Conn — обёртка вокруг ресурса с метаданными (время создания, время последнего использования).

---

### 007 — sync.Pool & Object Reuse

**Концепт:** `sync.Pool` для переиспользования временных объектов, уменьшение аллокаций.

`sync.Pool` — не пул соединений. Это кэш переиспользуемых объектов: буферов, структур, промежуточных результатов. GC может очистить его в любой момент. Ценность — снижение нагрузки на аллокатор и GC на горячих путях. В контексте connection pool — переиспользование буферов для чтения/записи.

**Результат:** интеграция `sync.Pool` для буферов в connection pool. Бенчмарк: с sync.Pool и без — сравнение аллокаций.

---

### 008 — Bulkhead Pattern

**Концепт:** изоляция нагрузки — разные категории трафика получают отдельные пулы ресурсов.

Без bulkhead: один тяжёлый клиент исчерпывает все соединения, лёгкие клиенты стоят в очереди. С bulkhead: пул разделён на секции (критичный трафик — 5 слотов, обычный — 10, фоновый — 3). Каждая секция — отдельный семафор. Перегрузка в одной секции не влияет на остальные.

**Результат:** `BulkheadPool` с именованными секциями и независимыми лимитами.

---

### 009 — Singleflight & Thundering Herd

**Концепт:** дедупликация одинаковых запросов через `singleflight`, защита от thundering herd.

Кэш инвалидировался. 100 горутин одновременно запрашивают одни и те же данные. Без singleflight — 100 запросов к базе. С singleflight — один запрос, 99 ждут его результат. `golang.org/x/sync/singleflight` — стандартное решение. `golang.org/x/sync/errgroup` — параллельные операции с отменой при первой ошибке.

**Результат:** интеграция singleflight в connection pool для дедупликации. `errgroup` для параллельного health check соединений.

---

### 010 — Production Assembly & Stress Test

**Концепт:** финальная сборка, stress test, профилирование, метрики пула.

Все компоненты объединяются в production-ready Connection Pool Manager: семафор для ограничения, rate limiter для throttling, bulkhead для изоляции, singleflight для дедупликации. Stress test с сотнями горутин, race detector, pprof для профилирования. Метрики: active connections, idle connections, wait time, acquire latency.

**Результат:** `connpool` пакет готовый к использованию в проектах 4–10.

## Как работать

Формат тот же что в проекте 1 и workerpool-practice.

Для каждого модуля генерируется `condition.md` (задание, требования, эксперимент на разрушение, вопросы до кода) и `theory.md` (теория концепта с примерами и разбором). Потом пишется код, отправляется с выводом программы на ревью. После финализации кода — QA файл на закрепление.

Контрольный вопрос перед переходом к следующему модулю: могу ли я объяснить концепт устно, воспроизвести баг из эксперимента и ответить на все вопросы QA файла не подглядывая?

10 модулей × 3–5 дней = 5–7 недель. Модули 003 (weighted semaphore), 006 (pool core) и 010 (stress test) могут потребовать больше времени. Глубина важнее скорости.
